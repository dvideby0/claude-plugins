# Architecture Audit

Read `${CLAUDE_PLUGIN_ROOT}/phases/shared-preamble.md` for critical rules.

This audit focuses on architectural health — module structure, coupling,
risk distribution, and design patterns.

---

## Step 1: Explain to User

> **What `/audit-arch` does:**
>
> Analyzes your project's architecture:
> - God module detection (too complex, too many dependents)
> - Layering violations (UI importing from DB, etc.)
> - Coupling analysis with risk scoring
> - Bus factor per module (git contributor concentration)
> - Module responsibility analysis
> - Risk-ranked modules for prioritization
>
> **Estimated time:** 3-5 minutes
> **Output:** `sdlc-audit/reports/ARCHITECTURE_REVIEW.md`
>
> Does NOT modify any of your files.

Ask the user to confirm before proceeding.

## Step 2: Discovery + Pre-Analysis

If `sdlc-audit/data/detection.json` does not exist, read and execute
`${CLAUDE_PLUGIN_ROOT}/phases/discovery.md`.

Run pre-analysis for code metrics and git history:
- Step 0h (Code Metrics) from `${CLAUDE_PLUGIN_ROOT}/phases/pre-analysis.md`
- Step 0i (Git History) from `${CLAUDE_PLUGIN_ROOT}/phases/pre-analysis.md`

Report: **[1/4] Discovery and metrics complete**

## Step 3: Module Analysis

If `sdlc-audit/modules/*.json` already exist from a previous audit, reuse them.

If not, read and execute `${CLAUDE_PLUGIN_ROOT}/phases/deep-analysis.md` to
spawn sub-agents for full module analysis. This is needed because architecture
review requires understanding module purpose, complexity, and internal structure.

Report: **[2/4] Module analysis complete** — [N] modules analyzed

## Step 4: Architecture Analysis

### 4a: Dependency Graph + Risk Scoring

```bash
bash ${CLAUDE_PLUGIN_ROOT}/scripts/build-dep-graph.sh .
bash ${CLAUDE_PLUGIN_ROOT}/scripts/compute-risk-scores.sh .
```

### 4b: Architecture Agent

Spawn the Architecture agent from `${CLAUDE_PLUGIN_ROOT}/phases/cross-module.md`
(the "Agent: Architecture" section). This agent reads module JSONs, dependency
data, and risk scores.

Report: **[3/4] Architecture analysis complete**

## Step 5: Generate Architecture Report

Write `sdlc-audit/reports/ARCHITECTURE_REVIEW.md`:

```markdown
# Architecture Review

*Generated by repo-audit (architecture mode)*

## Risk Dashboard

| Module | Risk Score | Category | Blast Radius | Complexity | Safety Net |
|--------|-----------|----------|--------------|------------|------------|
[Top 10 modules by risk score from risk-scores.json]

## God Modules
[Modules with high complexity + high fan-in + many files]
[For each: why it's a god module, what to extract, suggested decomposition]

## Layering Violations
[UI -> DB, tests -> production, infra mixed with business logic]
[For each: which modules, what the violation is, how to fix]

## Coupling Analysis
[Tightly coupled module pairs]
[Circular dependencies with severity assessment]
[Decoupling suggestions]

## Module Responsibilities
[For each major module: stated purpose, actual responsibility scope]
[Flag modules doing too many things]

## Bus Factor
[Modules with only 1 contributor — single points of failure]
[Data from git-busfactor.txt]

## Dependency Structure
[Module dependency graph summary]
[Hub modules — healthy vs concerning]
[Orphan modules — used but not imported by anything]

## Risk Interpretation
[For each high-risk module: why it's risky, recommended action]
[Prioritized list of what to tackle first]

## Recommendations
1. **Immediate** — [critical architecture issues to fix now]
2. **Strategic** — [architecture improvements for next quarter]
3. **Long-term** — [major refactoring suggestions]

---
*Generated by repo-audit (architecture mode)*
```

Report: **[4/4] Architecture report complete**

Present risk dashboard summary to user.
