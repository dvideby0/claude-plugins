# Test Coverage Audit

Read `${CLAUDE_PLUGIN_ROOT}/phases/shared-preamble.md` for critical rules.

This audit analyzes test coverage across your codebase, identifying gaps
and prioritizing what to test next.

---

## Step 1: Explain to User

> **What `/audit-coverage` does:**
>
> Analyzes test coverage across your project:
> - Per-module coverage assessment (full / partial / none)
> - Critical untested paths (high-risk code with no tests)
> - Test quality analysis (shallow tests, missing edge cases)
> - Testing pattern gaps and recommendations
> - Risk-weighted prioritization of what to test next
>
> **Estimated time:** 2-3 minutes
> **Output:** `sdlc-audit/reports/TEST_COVERAGE_MAP.md`
>
> Does NOT modify any of your files.

Ask the user to confirm before proceeding.

## Step 2: Discovery

If `sdlc-audit/data/detection.json` does not exist, read and execute
`${CLAUDE_PLUGIN_ROOT}/phases/discovery.md`.

Report: **[1/4] Discovery complete**

## Step 3: Module Analysis

If `sdlc-audit/modules/*.json` already exist from a previous audit, reuse them.

If not, read and execute `${CLAUDE_PLUGIN_ROOT}/phases/deep-analysis.md` to
spawn sub-agents for full module analysis. Coverage analysis requires the
`test_coverage`, `has_tests`, and issue data from each module.

Report: **[2/4] Module analysis complete**

## Step 4: Coverage Analysis

### 4a: Risk Scoring (for prioritization)

If `sdlc-audit/data/risk-scores.json` does not exist:

```bash
bash ${CLAUDE_PLUGIN_ROOT}/scripts/build-dep-graph.sh .
bash ${CLAUDE_PLUGIN_ROOT}/scripts/compute-risk-scores.sh .
```

### 4b: Coverage Agent

Spawn the Coverage agent from `${CLAUDE_PLUGIN_ROOT}/phases/cross-module.md`
(the "Agent: Test & Documentation Coverage" section).

Report: **[3/4] Coverage analysis complete**

## Step 5: Generate Coverage Report

Run the test coverage assembly script:

```bash
bash ${CLAUDE_PLUGIN_ROOT}/scripts/assemble-test-coverage.sh .
```

If the script ran, read and enhance the report with the coverage agent's findings.
If jq was not available, generate the full report from module JSONs.

Ensure the report includes:

```markdown

*Generated by repo-audit (coverage mode)*

## Coverage Summary

| Status | Count | Modules |
|--------|-------|---------|
| Full coverage | [N] | [list] |

| N/A (config, docs) | [N] | [list] |

## Critical Gaps (Test These First)

[Modules with high risk score AND no/partial test coverage]
[Sorted by risk score descending — highest priority first]

| Module | Risk Score | Coverage | Why It Matters |

[from risk-scores.json + coverage data]

## Per-Module Assessment

### [module name]
- **Coverage:** full | partial | none
- **Test types present:** unit / integration / e2e
- **Test types missing:** [list]
```
